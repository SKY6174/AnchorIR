# 🔒 AnchorIR 프로젝트 보안 가드레일 (GUARDRAILS.md)

이 문서는 AnchorIR 프로젝트의 보안 성능을 유지 및 강화하고, 사용자 개인정보와 핵심 데이터를 안전하게 보호하기 위해 에이전트와 개발자가 반드시 준수해야 하는 보안 가드레일입니다.

---

## 1. Supabase 데이터베이스 접근 제어 (RLS 필수 적용)
- **모든 Supabase 테이블은 Row Level Security (RLS)가 의무적으로 활성화되어야 합니다.**
  - RLS가 설정되지 않은 테이블은 API 주소와 `anon` 키만 알면 외부에서 무단으로 데이터를 조회, 수정, 삭제할 수 있어 매우 위험합니다.
  - 새 테이블 생성 시 반드시 아래 SQL을 실행하여 RLS를 활성화해야 합니다:
    ```sql
    ALTER TABLE 테이블명 ENABLE ROW LEVEL SECURITY;
    ```
- **정책 (Policies) 수립**:
  - 권한이 없는 사용자가 다른 사람의 데이터나 전체 데이터에 접근하지 못하도록 세부 SELECT, INSERT, UPDATE, DELETE 정책을 명시해야 합니다.
  - 로그인한 사용자만 본인의 데이터를 볼 수 있게 제한하는 정책 예시:
    ```sql
    CREATE POLICY "Users can only view their own items" 
    ON 테이블명 FOR SELECT 
    USING (auth.uid() = user_id);
    ```

## 2. API 키 및 자격 증명 관리 (프론트엔드 노출 금지)
- **프론트엔드 코드 내에 데이터베이스 비밀번호나 Supabase `service_role` 키를 절대 하드코딩하거나 작성하지 마십시오.**
  - `service_role` 키는 RLS를 우회하여 데이터베이스의 모든 테이블을 무제한으로 제어할 수 있는 관리자 권한 키입니다.
  - 이 키가 프론트엔드 클라이언트 빌드(JS 파일)에 포함되어 배포되면 빌드 파일을 파싱하여 키를 탈취당할 수 있습니다.
- **클라이언트 환경의 키 규칙**:
  - 클라이언트 브라우저(`src/` 폴더 내 코드)에서는 오직 RLS 정책이 적용되는 안전한 **`anon` (anonymous) public 키**만 사용해야 합니다.
  - 관리자용 배치 작업이나 RLS 우회가 필요한 복잡한 트랜잭션 로직은 프론트엔드가 아닌 **Supabase Edge Functions** 또는 백엔드 API 서버 내에서 `service_role` 키를 불러와 안전하게 처리한 후 결과만 반환해야 합니다.
- **환경 변수 파일 관리**:
  - `.env` 및 `.env.local` 등 모든 환경변수 설정 파일은 `.gitignore`에 의무적으로 등록하여 Git 원격 저장소(GitHub 등)에 절대 업로드되지 않도록 방지합니다.

---

## 3. 민감 데이터 암호화 및 개인정보 보호
- **학생 정보, 교직원 정보, 비밀번호 등 민감 데이터 평문 저장 금지**:
  - 사용자 비밀번호는 Supabase Auth가 제공하는 해싱 시스템을 이용해 자동으로 보호되나, 그 외에 데이터베이스에 개별적으로 저장하는 학생 정보나 계정 관련 민감 값들은 평문으로 저장해서는 안 됩니다.
  - 데이터베이스에 저장하기 전에 암호화 로직(예: CryptoJS 라이브러리를 활용한 AES-256 암호화 또는 Supabase 내의 `pgcrypto` 확장 모듈을 사용한 SQL단 암호화)을 거친 후 암호문 상태로 저장해야 합니다.

---

## 4. 에이전트 개발 협업 및 보안 검사 규칙
- 모든 신규 SQL 마이그레이션 파일이나 테이블 변경 쿼리는 이 가이드라인의 RLS 적용 가이드를 충족하는지 검증받아야 합니다.
- 코드 작성 완료 및 커밋 전에 데이터베이스 키나 비밀번호가 평문으로 소스 코드 내에 남아있지 않은지 정적 코드 검사를 상시 수행해야 합니다.
